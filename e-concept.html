The e- Concept

Making our own framework

Making our own framework for HTML/CSS is not so weird or hard, really. I think that some of the reasons why we don’t do it are:

  1. There are vendors’ solutions tested and with great support.
  2. We think that we won’t use in another project the stuff we are making now.
  3. We don’t want to stop and spend time organization this stuff to use it later (we can always copy/paste).
  4. We don’t have a workflow that allows us to do it in an organized and easy way.

These are good reasons. Now, let’s break them.

  1. Of course there are better solutions out there. The makers of these solutions worked very hard for long periods of time to create these tools. But I’m not saying that we should stop using them, I’m saying that we can choose and use what we like and need from these solutions in an organized and systematic way (not just copy/paste something that just "works"). One fundamental step in becoming a good developer is to see others people’s code, and this is a great way to do it.

    Many of the vendors’ solutions are made in a modular way where you take just what you want. For example, let’s say you like the Bootsrap grid system. You don’t need to have the whole framework to use the grid, just insert its modules in your project. But after using them a few times, you might find yourself missing some features and overriding some parts of the grid system. So, why not modifying them permanently (giving them credit) and made them part of ECSS. So, next time you will have an e-grid system that feeds your needs.

  2. We might will use that code later, you know it.

  3. I know it’s hard to stop, especially when we have too much work, but we can take benefits from doing it. We just need to find an easy and fast way to do it.

  4. So, lets make that wotkflow!

Heading

You may find yourself saying

"I usually reuse the code I do, without any methodology, so I don't see the purpose of making any utilities or framework."

You will be amazed by the many things you don’t re-utilize that you could if you just follow a methodology. It’s like when we set the goal of going to the gym every day at a specific hour, let’s say 6 a.m. After a while, you find yourself going more to the gym than when you just go at any random hour. This is because you are following a pattern.

About reusing the code you write (and using the code you see) in HTML/CSS, the e- concept is my suggestion. Find patterns in your code and put them in a place that tells you they are patterns.

Example

      // Project x, somewhere in the code...
      .a-big-button {
          // code
          display: block;
          width: 100%;
          // code
          // code
          // code
      }

      .some-other-button {
          // code
          // code
          display: block;
          width: 100%;
          // code    
      }

Becasue you find yourself repeating width: 100%; and display: block; many times, you decide to create a btn--full modifier that you can place in the markup or extend in Sass. That is clever. Then you finish the module or the project, and forget about the btn--full modifier. If you need something like that in the future, you already know how to do it, or you can copy/paste it, or you can use a framework with something similar. Right? Absolutely right, but another option is to set the btn--full modifier in a place you could have easy access in the future. 

        |– elements/ 
        |   |– _button.scss
           
                .e-btn--full {
                   width: 100%;
                   // code     
                 }

You know buttons are going to be needed in your next project, so bring the button.scss element and the e-btn--full modifier will be ready to be used. You know this class is part of ECSS because of the e- prefix. You treat e- definitions differently than projects definitions, since you don’t modify them directly (more of that below) therefore you will know, or the rest of your team will know, that an element is not broken.

Modifying an e- element in a project

I suggest one of these two approaches for this.

1- A file

Create a file with the a different prefix name (let’s say eo-) and override the defaults (the override variables file of Bootsrap is a good example).

        |– settings/ 
        |   |– _eo-global.scss          # Here you override the defaults of e-global    
        |   |– _e-global.scss           # evolve CSS global settings
        |   |– _global.scss             # Your project globals      
        |

Take in count that you need to import before the file that overrides.

2- Do it in the app.scss (the file where you import all the partials)

This technique I saw it in inuit css and I liked it very much. The idea is that you set the override just before importing the file

       
        /**
         * #SETTINGS
         */
        @import "settings/etc"; 

        $e-font-size:         14px;
        $e-line-height:       21px; 
        @import "settings/e-global";
        @import "settings/etc";

You leave a line after the previous import (for legibility), then override the variables of the import you want.

What I like of this approach is that you can easily see in one file what is difference with the way you are used to work (e.g.: you are using a base font-size of 14px this time, not 16px as you are used to it).

Combining ECSS with other frameworks

Let’s say that you want to use ECSS, and that you want to use Foundation grid system, but you don’t want to use anything else from it. So you include Foundation grid system in the vendors’ module.

        /**
         * #VENDORS
         */
        --- Import the main file ---
        @import "vendors/foundation/scss/app";

        --- And then, comment in that file the @import "foundation";(which include all the components), 
        and uncomment the grid import ---
        // @import "foundation";
        @import
        "foundation/components/grid";
        //   "foundation/components/alert-boxes",
        //    etc...

Or, use the exclude system of Foundation

        /**
         * #VENDORS
         */
         --- Set to false the global include variable in Foundation -settings.scss and set 
        to true the grid include variable ---
        $include-html-classes: false;
        ...
        $include-html-grid-classes:true;

This approach will import also all the mixins of the other foundation components (which you can use of course), without printing any CSS. The reason of this is because Foundation components mixins are in the same file where the CSS printing is (as difference to Bootstrap, that separates components mixins in their own file). So, if you don’t include the component, you won’t have its mixins.

Making your own grid system

If you want to have your own grid system, just make it an ECSS element.

        |– elements/ 
        |   |– _e-grid.scss           # Your grid system in your framework    
        |   ...                       # Etc…        
        |

Combining

Let’s say that you want to use ECSS and Bootstrap buttons. You want to use some features as the .btn-success Bootstrap class, and don’t want to make it yourself. So, what you do is to stop importing the e-button element and import Bootstrap button component.

        /**
         * #VENDORS
         */
         @import "vendors/bootstrap/bootsrap.scss"; 
         

        /**
         * #ELEMENTS
         */  
        // @import "elements/button";
        @import "elements/other-element";

But wait. You don’t have to remove the ECSS button element if you don’t want to. Let’s say that you want to use some modifiers that you have in e-button.scss, as you use the .btn-success Bootstrap class. You can mix them. There will be not conflict between the button classes because of the e- prefix. Then, you can do something like this.

<button class="btn  btn-success  e-btn--full"></button>

Bootstrap already has a full button modifier, so you don’t need to do something like this (unless you prefer to use your full button modifier instead of Bootstrap's). Anyway, it’s just an example so you can see the possibilities that this approach brings.

In the same way, you can extend Bootrap classes and use its mixins inside your projects code (don’t do it inside ECSS code because then you make it dependent to Bootstrap). As an example, you can extend Bootstrap full button modifier.

        |– component/ 
        |   |– _button.scss
                |
                |- .btn--full {
                     @extend .btn-block;
                     //more code     
                   }

But instead of using this approach, with the goal of reducing the mix of your code with Bootstrap' code, I suggest that you take some minutes to read the component in question (in this case the Bootstrap button component), understand how it works, and take just what you need and put it in your project button component. And later on, if you find yourself using this code in many scenarios, make it part of ECSS.

Passing code from project to ECSS

If you believe that something could be passed from your project to ECSS, you can do it in many ways. This is just one idea. For files, you can have beta files, and use an especial prefix for them, like "eb-" for example. And after you see that this work fits you well in another projects, you can make it part of the defaults of ECSS. For code, use a comment tag as TO-EVOLVE. Try many approaches to see which fits your need. If you want to be really serious with ECSS, you can have it in its own private or public project, as a github project.

Final thoungts

I hope you like this easy methodology. Remember, ECSS is not some outsiders weird code you don’t know and have never seen, it’s just your utilities, the code you like and use in an organized way.
