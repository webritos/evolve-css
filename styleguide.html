A modular Front-end Styleguide and Workflow

Basics

A small intro to the e- concept

ECSS could easily become a framework. It brings defined elements that can be used from the begging of the setup (as a font-size mixin for example). But the idea behind it is not that you stay away from that definitions and don’t touch them, neither that you only touch a override of the settings variables, as the Bootstrap/Foundation approach. The idea is that you create your own framework that suits your needs.

Usually, developers stay away from the framework that they are using because they bring so much stuff that we don’t know how everything is related, and we are afraid that if we touch something we could break something else. And of course, we could study it, but that takes time and maybe we won’t use it again in a long period of time, or we just feel that we don’t need to.

In ECSS, the variables, mixins, files and rest elements that are defined to be part of its defaults, begin with the prefix "e-". Feel free to change these values, and create more if you wish. My suggestion is that when you see some pattern in you code, while going from one project to the other, set those patterns as part of ECSS. Just put it in its place (if is a mixins in the e-mixin.scss file, for example) and add them the e- prefix.

So, now you know that if you see a file called "_e-mixins.scss", is the partial where the framework mixins lies. And if you see a "_mixins.scss" file (no "e-" prefix) is the project mixins holder. Keep working in your daily busy life, using ECSS in many projects, and when you find a pattern (let’s say you see yourself defining a mixin often, so create a e- version for that mixin and next time you will have it there and you will gain speed (specially because you don’t have to remember it in the middle of some task related).

If you are not interested in creating and maintaining a framework, you don't have to. Just take what you like from ECSS and leave what you don’t (as I did with the projects mentioned previously). If you like to know more about the "e-" idea, there is a section that explains in more detail, otherwise just continue reading.

Due to the concept of ECSS, you don’t need to wait for updates (but there’ll be more versions).

File structure

This is the way I’m organizing my files. You can change however you want, but if you decide to follow this approach, there are some things (described below) to be considered. Many of these files are just examples, so don’t take them literally. Right now at github, only the files with an * have content.

One of the aspects that you will see is that the e- files are not in a separate folder, the project files and ECSS files lie in the same place. The idea of this is that ECSS’s code and the projects’ code are more related, and that you can easily work in both of them. This is because ECSS is not some outsiders weird code you don’t know and have never seen, it’s just your utilities, the code you like and use in an organized way.

As you see, is a modular approach. The modules must be included in this order, but the order of the files inside them doesn’t matter. The explanation of each section below.

Vendors

Put your vendors’ assets here. There is an override folder inside where you can override anything without touch in the vendors’ files.

Settings

Is where your global variables are set. Base font-size, global colors, media-queries breakpoints, etc. This way we reduce the amount of magic numbers and we reduce the amount of different values for an aspect. It's common to end with projects with 10 small variations of the same color. Anyway, you could define and use local variables inside any module.

Tools

The area for the functions, mixins and aliases. ECSS’s defaults are in _e-functions.scss, _e-mixins.scss and _e-aliases.scss. You can put the ones of your project in the respective file without the "e-" prefix.

Generic

The idea of the generic module is to put elements that set global-base styles that can be a specific implementation (like _box-sizing.scss, that set the " box-sizing: border-box;" for all elements) as a global set of definitions (as resets or normalize). If you want to define something as the vertical rhythm of the site (a definition that is a base statement and affects different HTML elements that are not related, the generic module is the place to go. (e.g.: forms are group of related elements, so they go in the elements module).

        // generic
        html {
            -webkit-box-sizing: border-box;
               -moz-box-sizing: border-box;
                    box-sizing: border-box;
        }
        * {
            &,
            &:before,
            &:after {
                -webkit-box-sizing: inherit;
                   -moz-box-sizing: inherit;
                        box-sizing: inherit;
                }
        }

Elements

The Elements module is where the base definition of HTML elements, or related group of elements, are set (list, buttons, etc). Each element has its file (or group of elements related). In methodologies as SMACSS, there is a base module where you only style tags and don’t style classes, so you end with the basic/default look of elements and leave the classes for the components module.

In ECSS, tags and classes can my styled in the elements module. This idea emerges after my struggle deciding what to do with some elements after seeing other architecture methodologies. In some approaches, buttons are styled first in a "base" module, then in an "object" module and finally in the "components" module. The idea behind this is that in "base", things as "cursor: pointer;" are defined, and then in "objects", they make some default button, as some default modifiers (like .btn--full), and finally they style the button in "components". For me, this approach is too disperse.

I like the idea of "extending" an object, so in ECSS you define have the elements module for setting the defaults of buttons, add some modifiers if you wish, and then extend the button element with some UI styles (and more modifiers if you wish). You will understand better after reading the objects and components descriptions below.

      // base of the button element. The tag "button" dosen't has to be 
      // included if you dont want, the class is the real protagonist.
      button, .btn {
        display: inline-block; 
        vertical-align: middle; 
        font: inherit; 
        text-align: center; 
        etc...
      }

We can see that those CSS statements can be used by the button tag as by its emulation with the .btn class. This is why, for me, the element module is similar to an object that can be extended by a component.

Objects

An object is a small structure, where you can create something that is not an HTML regular group of elements (like a .container class) by combining elements together (like the "quarks->atoms" approach of atomic design). The best example I can think of is combing elements in a reusable and adaptable small structure is the media object, the golden child of the OOCSS. Below, a container and a box object.

      // object
      .container { 
        width: 980px;
        margin-right: auto;
        margin-left:  auto;
      }

      // object
      .box { 
        display: block; 
        padding: $someValue;
      }

Components

Is where you will define the UI of the elements, and is where you will spend the most of your time.

      // component
      .btn {
        color: $e-btn-text-color;
        font-weight: bold;
        font-size: 17px;
        border-style: solid;
        border-width: 1px;
        border-color: $e-btn-border-color;
        border-radius: 2px;
        background-color: $e-btn-bg-color
        box-shadow: 0px 1px 2px 0px rgba(22, 20, 18, 0.196);      
      }

Components can extend from the element module as from the object module, and can also define its own structure or element that is not defined anywhere else. Here, you can set modifiers and sub components, and override anything from their parents.

---- Pause ----

How you went back there with the generic/elements/objects specifications? Some tips to have it clearer:

  • What you define in "generic" is not the style of specific HTML elements (or group of elements related together). What you have in "elements" are styles of diverse HTML elements which are not related together (as a reset). You can think of this as “not specific base” module. The Elements will be then a specific base module.
  • Components can extend from elements as from objects. You can use the same class in a component that you use in its parent (element or object).
  • You only use tag elements in "generic" and in "elements". You don’t use them in "objects" or "components", here you only use classes.

Structures

A structure is just a component that it has grow too big to be really modular; that means that its connected to others elements/objects (e.g.: is inside of some parent container or group of containers). Why an area for this type of component if there is a component area? The idea of a component is that is modular. If you move it to another project (with its abstraction if it has one, an element or an object) it will look and behave in the same way. But if the component doesn’t do this, it can become hard to move components because you won’t know if it will work or not.

How to know when a component becomes a structure? Don't try to anticipate this, just work as usual in the components module, and when you spot some component that is not modular, move it to structures. I took the idea from MVCSS, where they explain it like this:

"In the end, limiting scope and minimizing dependencies both affect portability, which is usually our biggest concern when classifying modules. If you can easily tweak a few lines of CSS inside a module and use it between projects, that means it’s reasonably portable, and you have solid grounds for classifying it as a Component. On the other hand, if try to port a module and find yourself rewriting a significant amount of code, then the module is likely better classified as a Structure."

Pages

If for some reason you don’t find a correct place for an exclusive styke, that only happen in a page and it won’t repeat, use this area. Anyway, It’s better to use the elements/objects/components approach, and use some helpers classes in the markup if you need to twist something so small it doesn’t deserve the time that takes to establish its place and name it.

Themes

Are group of styles that change drastically the look of the site. It can be a Christmas theme for example, or a "crazy offer day" theme. Usually, you don’t use them and the folder is empty.

Trumps

Helper classes. From margin definitions to our classical .hide. most of them has an !important, to prevent CSS specify issues. But using !impotant isn’t evil? No. The problem is to use it in an unorganized environment, where it’s hard to tell what is been overridden. Many frameworks use them. The concept of a helper class as .hide is that if you use it, it’s because you want that element to be hidden, so .hide should trumps above others.

Inbox

A place where you can code without worrying about rules. Is an area where you can code if you are not sure where to place a statement. Later on, you or the person in charge of the maintenance of the architecture would refactor it and place it in a better place.

Naming conventions

Clasess

This is the naming convention for classes. It combine BEM, SMACSS an others ideas. 

          // Naming convention
          .component-name
          .component-name--modifier-name
          .component-name__subcomponent-name
          .component-name__subcomponent-name--modifier-name

Examples

        <nav class="primary-nav  primary-nav--hightlight">
          <ul class="primary-nav__list">
            <li class="primary-nav__item  primary-nav__item--shadow">
                <button class="btn  btn--full  btn--large"></button>
            </li>
          </ul>
        </nav>

Many people find this naming convention ugly at first. If you are unfamiliar with BEM and similar, and you don’t understand what I’m doing, take your time and study it. Meanwhile, read the CSS guidelines.

I use BEM naming convention for Sass elements too:

        // Variables
        $element-text-color:        value;  
        $element--primary:          value;  // modyfier     
        $element--big:              value;  // modyfier    

        // Mixins
        @mixin btn-style {}

More:

Use a js-prefix for javascript hooks (dont style them in CSS to keep them independient to presentation)

        <button class="btn  js-btn" />

Use an is- prefix for classes related to a state. More here (For more about this, I suggest you to get Jonathan Snook great book SMACSS, about many of the topics explained in ECSS).

        <div class="element  is-collapsed  e-is-marked" />

        .element.is-collapsed { }

Use a has-prefix for denotating context situation. More here

Use e- prefix for ECSS default elements.

Standars

The way to go

Many of the ideas of this section come from the Harry Roberts CSS Guidelines, so I suggest you to read it also. The differences of my implementation with his are minimum (he uses spaces for indentation but I prefer tabs, for example, (one debate about it)).

Here are some principals that I try to implement (I swear I try as much I can). You don’t have to follow them, but at least is good that you have them in account.

  • Multi-line CSS

  • Meaningful use of white space

  • Titling all the files and give meaningful information in descriptions

  • I'm separating multiple classes in HTML with two spaces

                <div class="carousel  box">

  • I'm grouping related classes in the HTML in square brackets ([ and ]) to help readability. Notice that the brackets must be spaced out. More here: Grouping related classes in your markup. 

              <a class="[ btn btn--large ]  page-head__link">

  • I’m using a data-ui-component attribute to give semantic name to relevant HTML elements. More here.

  • I don’t use id to style elements, that way is easy to manage CSS's specificity. More here

  • I do use !important for helpers classes, and no, it’s not a bad practice.

  • I try to follow as much as possible the Single-direction Margin Declaration, which means that I try to don’t use margin-top and margin-right in any element (hard but convenient).

Semantics

The debate about the importance of semantics in front-end development is a big and old one. For me, writing semantics HTML's tags is the correct way to go (I even enjoy when trying to decide what tag to use in a concrete scenario, even if a have to investigate some minutes). So, if the element is a header, use <header> instead of <div>

But when it’s about classes, I think differently. I almost don’t see a real reason to use a semantic class name. It doesn’t affect SEO and screen-readers don’t read them. About helping others to understand our code, in what statics web concern, I would like to cite Harry Roberts on this:

Using a class name to describe content is redundant because content describes itself.

I think it’s more useful to style using classes in a way that they become more reusable, even if they are presentational. More about this:

Fragments of good ideas

These are some excerpts from Code smells in CSS, a post that I really recommend you to read.

About Layout

"All components you build should be left totally free of widths; they should always remain fluid and their widths should be governed by a parent/grid system.

Heights should never be be applied to elements. Heights should only be applied to things which had dimensions before they entered the site (i.e. images and sprites). Never ever set heights on ps, uls, divs, anything. You can often achieve the desired effect with line-height which is far more flexible.

Grid systems should be thought of as shelves. They contain content but are not content in themselves. You put up your shelves then fill them with your stuff. By setting up our grids separately to our components you can move components around a lot more easily than if they had dimensions applied to them; this makes our front-ends a lot more adaptable and quick to work with.

You should never apply any styles to a grid item, they are for layout purposes only. Apply styling to content inside a grid item. Never, under any circumstances, apply box-model properties to a grid item."

Magic numbers and absolutes

A magic number is a number which is used because ‘it just works’. These are bad because they rarely work for any real reason and are not usually very futureproof or flexible/forgiving. They tend to fix symptoms and not problems.

For example, using .dropdown-nav li:hover ul { top: 37px; } to move a dropdown to the bottom of the nav on hover is bad, as 37px is a magic number. 37px only works here because in this particular scenario the .dropdown-nav happens to be 37px tall.

Instead you should use .dropdown-nav li:hover ul { top: 100%; } which means no matter how tall the .dropdown-nav gets, the dropdown will always sit 100% from the top.

Every time you hard code a number think twice; if you can avoid it by using keywords or ‘aliases’ (i.e. top: 100% to mean ‘all the way from the top’) or—even better—no measurements at all then you probably should.

Every hard-coded measurement you set is a commitment you might not necessarily want to keep.

Comments

I like the idea of having defined comments in order of decreasing importance. This is the standard I like and use.

        /***********************************************\

           #FIRST LEVEL - For naming

        \***********************************************/


        /*----------------------------------------------*\
            Second level - For naming
        \*----------------------------------------------*/


        /*  Third level - For naming
        -------------------------------------*/


        /* ----- Fourth Level - For naming ----- */


        // Fifth level - General purpose



        /**
         * High-level - For large comments that document entire module
         *
         * 1) Information about the module
         * 2) More information
         */ 


        /**
         * Low-level - For on specific declarations
         *
         * 1. Information about the declararion
         * 2. Information about the declararion
         */

        .page-head--masthead {
          color: #000;        /* [1] */
          text-align: center; /* [2] */
        }

Your own Documentation

There are many useful tools to automatize the process of creating a front-end style guide. It’s in my plans to continue extending ECSS and I will probably make some helper for it. You can use one of these tools or you can edit this documentation so it fits to your needs.

I strongly recommend that you make a style guide of the projects you work on. I know you might have so much work, but I suggest that when you have some free time, that you or your team take 3 days to study the concepts covered on this doc, specially concepts as OOCSS and BEM. In the same way, use that time to improve also your workflow. Get to know tools as emmet, Grunt, Bower and LiveReload. You won’t be making money in those three days, but your productivity and quality of code will improve drastically. I promise.
